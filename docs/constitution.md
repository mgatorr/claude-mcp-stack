# claude-mcp-stack Constitution

A curated, agent-first, open-source recipe that lets a Claude Code agent install vetted
finance/research MCP servers into Claude Desktop/Cowork and Claude Code. These principles
are non-negotiable and govern every contribution. The approved design lives at
`docs/superpowers/specs/2026-05-30-claude-mcp-stack-design.md`; this constitution outranks
it where they conflict.

## Core Principles

### I. Security & Privacy First (NON-NEGOTIABLE)

No secret or personal datum may ever enter the repository or its git history.

- Templates and examples MUST contain placeholders only (e.g. `${TWELVEDATA_API_KEY}`,
  `"Jane Doe jane@example.com"`, `$HOME`); they MUST NOT be generated by copying a real
  user config.
- A pre-push secret scan MUST run before any publish, in two layers — a value-based
  denylist (known strings + generic regexes for home paths and UUIDs) AND a structural/
  entropy scanner — and MUST cover the working tree AND the full git history. Any hit
  blocks publishing.
- Real API keys, emails, names, account identifiers, device names, and absolute home paths
  are forbidden in tracked files. `.gitignore` MUST exclude real configs, `*.bak`, `.env`,
  and virtualenvs.

Rationale: the repo is public and handles credentials; a single leaked key or identity is
irreversible once in history.

### II. Agent-First, Human-Legible

The primary consumer is an AI agent; humans are a first-class secondary audience.

- Process and data MUST be split: an `AGENTS.md` playbook (how to install) and a
  machine-readable `registry.json` (what to install). Adding a server MUST be one
  `registry.json` entry plus one human doc — never a playbook rewrite.
- A polished human `README.md` MUST explain the project, the quickstart flow, and the
  security stance. `AGENTS.md` MUST open with a fixed preamble and define explicit success
  criteria and rollback rules so agent behavior is consistent.

Rationale: determinism for agents and clarity for humans are both required for trust.

### III. Idempotent & Safe Installs (NON-NEGOTIABLE)

Installation MUST never damage a user's existing configuration.

- Back up the target config before writing; write backups only to the OS config directory,
  never into the repo.
- Merge only the selected server keys; preserve every unrelated key (`preferences`, other
  servers, top-level fields) byte-for-byte.
- Ask before overwriting an existing server entry. On any verification failure, restore the
  backup and report — never leave a half-merged config.
- Secrets entered at install time MUST NOT be echoed to logs, transcripts, or any file
  other than the target config.

Rationale: users grant the agent write access to live config; safety must be structural,
not best-effort.

### IV. Recipe, Not a Vendor Dump

The repo orchestrates third-party servers; it does not republish them.

- Third-party MCP servers MUST be launched via `uvx`/`npx`; their source MUST NOT be copied
  into this repo.
- The one maintained component (the `mcp-yahoo-finance` fork) MUST preserve the upstream
  MIT license and copyright; modifications are documented, never relicensed. The registry
  MUST pin it to a tag/commit, not a moving branch.
- Upstream projects MUST be credited in the README.

Rationale: respecting licensing and attribution is both a legal and reputational baseline.

### V. Test-First for Real Code (NON-NEGOTIABLE)

Every executable artifact is built test-first.

- The registry validator, the config-merge helper, and the secret-scan script MUST each
  have a failing test written and observed to fail before implementation (Red-Green-Refactor).
- A merge test MUST prove unrelated config keys survive a merge unchanged.
- CI MUST run the validator, the test suite, and the secret scan on every push and PR.

Rationale: the install path touches user systems and credentials; correctness must be
demonstrated, not asserted.

### VI. Professional Presentation

The project must read as a serious, trustworthy open-source effort.

- It MUST ship `SECURITY.md`, `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, and `CHANGELOG.md`.
- The README MUST include a clear "not affiliated with, endorsed by, or sponsored by
  Anthropic" disclaimer, a tagline, a servers table, a "how it works" diagram, and an
  example agent transcript.
- All artifacts — including commit messages — MUST be written in English.

Rationale: presentation is the project's credibility and a reflection of the maintainer.

## Security & Privacy Requirements

- Public identity policy: commits use the maintainer's GitHub noreply email; the display
  name may appear, the personal email may not. `LICENSE` carries the name without an email.
- The secret scan denylist and the `.gitignore` rules are part of the security surface and
  MUST be reviewed whenever a new server, template, or example is added.
- Privileged install actions (package-manager installs of prerequisites) MUST be disclosed
  in the README so users can review `AGENTS.md` before running it.

## Development Workflow & Quality Gates

- Spec-Driven Development via Spec Kit: every feature flows spec → plan → tasks →
  implement, with artifacts under `specs/<NNN-feature>/`.
- A change is mergeable only when: tests pass, the validator passes, the secret scan is
  clean (working tree + history), and the relevant docs are updated.
- Frequent, small, English commits. Publishing to public remotes happens only after
  explicit maintainer approval and a clean secret scan.

## Governance

This constitution supersedes other practices for this repository. Amendments require a
documented change (what and why), a version bump per the policy below, and propagation to
any dependent templates and docs.

- Versioning policy (semantic): MAJOR for backward-incompatible principle removals or
  redefinitions; MINOR for a new principle or materially expanded guidance; PATCH for
  clarifications and non-semantic refinements.
- Compliance: every PR and review MUST verify adherence to these principles; deviations
  MUST be justified in writing or rejected.
